## 7. Technical Specifications

### 7.1 Configuration contract

Use the exact root variables from master section 23: `APP_ENV`, `FRONTEND_ORIGIN`, `VITE_API_BASE_URL`, `SQLITE_PATH`, `FILES_DIR`, Neo4j credentials, ShopAIKey settings, selected embedding settings, PDF/URL limits, and `TOOL_LOOP_LIMIT`. Docker Compose loads the root `.env`; separate frontend/backend environment files are forbidden.

### 7.2 SQLite ownership and schema

Create the master-defined tables: `attachments`, `candidate_profile`, `profile_drafts`, `job_preferences`, `job_posts`, `conversation`, `chat_messages`, `agent_runs`, `tool_executions`, `memory_facts`, and `graph_sync_outbox`. Use UUID identifiers except for singleton rows, UTC timestamps, explicit foreign keys, and JSON text/JSON columns validated at repository boundaries.

Status fields must remain independent:

```text
processing_status = received | processing | processed | failed
jd_quality = full | partial | unscorable
graph_sync_status = not_required | pending | synced | failed
record_status = active | ignored_duplicate
```

The initial migration must be safe for both an empty database and a persistent already-initialized volume. LangGraph checkpoint tables are not duplicated in application models; its SQLite checkpointer owns them later.

### 7.3 Attachment storage interface

```python
class AttachmentStorage(Protocol):
    async def stage(self, attachment_id: UUID, content: AsyncIterator[bytes]) -> StoredFile: ...
    async def promote(self, storage_path: str) -> str: ...
    async def delete(self, storage_path: str) -> None: ...
    async def open(self, storage_path: str) -> AsyncIterator[bytes]: ...
```

Paths are generated by the service, remain under `FILES_DIR`, and are never accepted directly from users. Plan 2 implements storage mechanics only; MIME, magic-byte, page-count, and profile replacement policy belong to Plan 4.

### 7.4 Graph schema bootstrap

Create unique constraints for `Candidate.id`, `Job.id`, `Skill.canonical_key`, and `JobFamily.canonical_key`. Create the cosine Job vector index using the selected Plan 1 dimension. Setup must be idempotent. Neo4j is derived and rebuildable; no canonical application state may exist only in Neo4j.

### 7.5 Outbox contracts

An outbox repository must support enqueue-in-transaction, claim bounded pending work, mark synced, and mark failed with attempt count. Operation identity is stable and replay-safe. Payloads contain SQLite IDs and canonical data, never filesystem paths, API keys, or raw documents. Continuous polling or a separate worker is prohibited.

### 7.6 Health boundary

`GET /api/health` reports overall status plus SQLite, filesystem, and Neo4j component states. It must not expose connection strings, paths outside a safe label, credentials, stack traces, or provider headers.

## 9. Verification & Testing Plan

- Backend lint and type-check pass with the pinned toolchain.
- Backend unit tests cover configuration validation, storage path containment, and outbox state transitions.
- Alembic upgrades a fresh SQLite file and reruns safely against an initialized persistent volume.
- Docker Compose starts frontend, backend, and Neo4j; frontend/backend bind to `127.0.0.1`.
- `/api/health` reports component states without secrets.
- Restarting Compose preserves SQLite/files and Neo4j volumes.
- Re-running Neo4j schema setup creates no duplicate constraints/indexes.
- A replayed outbox operation contract cannot create a second logical operation.
- No separate `.env`, worker service, public CRUD endpoint, or business feature from later phases exists.

Expected proof is a green local foundation command set and successful health/migration/schema-idempotency checks.
